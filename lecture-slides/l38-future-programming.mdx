## Every Generation Says "That's Not Real Programming"

<div style={{fontSize: '0.8em'}}>

**L1 callback:** Grace Hopper built one of the first compilers. The response?

> "I kept getting told that I couldn't do that, because a computer could only do arithmetic. They couldn't write programs."

</div>

<div className="fragment" style={{fontSize: '0.72em', marginTop: '0.8em'}}>

| Decade | New abstraction | "That's not real programming" |
|--------|----------------|-------------------------------|
| 1950s | Compilers | "Real programmers write machine code" |
| 1970s | Structured programming | "Real programmers use GOTO" |
| 1990s | Garbage collection | "Real programmers manage their own memory" |
| 2010s | Frameworks, low-code | "Real programmers write everything from scratch" |
| 2020s | AI coding assistants | "Real programmers write their own code" |

</div>

<p className="fragment" style={{fontSize: '0.85em', color: '#9370DB', marginTop: '0.5em'}}>
The pattern repeats: a tool automates part of the work. The definition of "programming" shifts upward to the work the tool <strong>cannot</strong> do.
</p>

<aside className="notes">
**This is the L1 bookend.** Walk through 2-3 rows. The GC row is fun — Java's GC was dismissed as a crutch by C/C++ programmers. Now it's the default.

Each row represents the same cycle: automation → skepticism → acceptance → the definition shifts upward.

**The question for 2026:** Where are we in this cycle? The answer depends on what the "work the tool can't do" turns out to be.

## The Hard Part Was Never the Typing

<p style={{fontSize: '0.85em', color: '#9370DB', marginTop: '0.3em'}}>
Fred Brooks (1986): AI is spectacularly effective <strong>above</strong> the waterline. Everything <strong>below</strong> it is what you are paid for.
</p>

<aside className="notes">
**This is THE key framework for the lecture.** The iceberg visual makes the point: the visible part (what AI automates) is small. The submerged mass (what remains) is enormous.

**Connect to Q2 from the poll:** "How many of you picked A (syntax/setup) as the hardest? That's above the waterline. How many picked B-E? That's below. AI is great above. It can't reach below."

**Fred Brooks, "No Silver Bullet" (1986):** Brooks argued that no single technology would deliver a 10x improvement in software productivity within a decade. He wrote this during a period of enormous hype around several technologies that were each being promoted as the breakthrough that would finally solve the "software crisis":

- **Ada** (1983) — the US Department of Defense spent billions developing a single programming language to replace the hundreds used across military projects. It was supposed to eliminate entire categories of bugs through strong typing and built-in concurrency. It helped, but didn't deliver 10x.
- **Object-oriented programming** — Smalltalk, then C++, then Java. OOP was marketed as a paradigm that would make software naturally modular, reusable, and maintainable. It improved things, but projects still ran over budget and behind schedule.
- **Expert systems / early AI** — the first AI hype cycle. Rule-based systems were going to automate programming itself. DARPA poured money into the "Fifth Generation" project. The AI winter followed when the promises didn't materialize.
- **CASE tools** (Computer-Aided Software Engineering) — graphical tools that would let you draw diagrams and auto-generate code. Basically the low-code of the 1980s. Generated terrible code and were largely abandoned.
- **Formal verification** — mathematically prove your program is correct. Promising in theory, impractical for real systems at the time (though making a comeback now — see the trends slide).

Brooks looked at all of these and said: they each attack **accidental complexity**, and they each work to some degree. But none can deliver 10x because most of the difficulty is **essential**:

- **Accidental complexity** = difficulty from tools and processes — boilerplate, syntax, API lookups, build config, manual memory management. Can be engineered away.
- **Essential complexity** = the irreducible difficulty of the problem itself — understanding requirements, making architectural tradeoffs, reasoning about edge cases, designing for change. If users need 30 features that interact in complex ways, that complexity is inherent in the problem, not the tools.

Brooks' prediction: tools can only deliver improvements proportional to the fraction of work that is accidental. Since most of the difficulty in software is essential (the problem, not the syntax), no tool can deliver an order-of-magnitude improvement overall. Every technology on that list genuinely improved things — and none delivered 10x. Sound familiar?

**Why *The Mythical Man-Month* (1975) still sells 10,000 copies a year:** At ICSE 2018 — the 40th anniversary of the conference — Brooks explained: "The book is not really about technology. It's really about people. The stage keeps changing, but the script doesn't change much." Communication overhead, conceptual integrity, the "second-system effect," the surgical team — all still relevant because they describe essential complexity in the human dimension.

**Brooks on AI (ICSE 2018):** He acknowledged AI would change software engineering, but restated his prediction: no single technology would deliver 10x in 10 years. "We'll check back in 2028." We're almost there.

**The twist students should feel:** Maybe AI HAS delivered 10x on accidental complexity. But we can't see it — because every time we eliminate one generation's accidental complexity, our expectations expand, bringing more essential complexity, which requires new tooling, which introduces new accidental complexity. The next slide makes this concrete.

## The Productivity Gain Is Real — It Is Just Invisible

<div style={{fontSize: '0.62em'}}>

| Era | Tools | What "a web app" meant |
|-----|-------|----------------------|
| 2000 | Hand-written HTML, CGI scripts | Static pages with a form |
| 2008 | Rails, Django, jQuery | Dynamic CRUD app with a database |
| 2015 | React, Docker, microservices | Real-time SPA with mobile support, CI/CD, monitoring |
| 2020 | Kubernetes, serverless, design systems | Multi-platform, accessible, globally distributed, auto-scaling |
| 2026 | AI-assisted development | All of the above, plus AI features, built by a smaller team |

</div>

<div className="fragment" style={{fontSize: '0.75em', marginTop: '0.5em'}}>

Each generation's tools made the previous generation's software **trivial to build.** But nobody used the tools to build last generation's software faster — they used them to build *this* generation's software at all.

</div>

<p className="fragment" style={{fontSize: '0.78em', color: '#9370DB', marginTop: '0.5em'}}>
**L36 (Jevons' paradox):** The 10x gain is real — it is invisible because the baseline keeps moving. 2026 has MORE accidental complexity than 2000, not less.
</p>

<aside className="notes">
**This is the mechanism that explains why Brooks keeps being right.** Productivity gains are absorbed by rising expectations.

The GA1 features students built — with AI, JavaFX, GitHub Actions, a team — would have been a senior capstone project 15 years ago. Are they "more productive"? Or doing something fundamentally more ambitious at the same pace?

## Feeling Fast Is Not the Same as Being Fast

<div style={{fontSize: '0.8em'}}>

Multiple controlled studies — they **disagree on speed** but agree on one thing:

</div>

<div style={{fontSize: '0.58em', marginTop: '0.5em'}}>

| Study | Year | N | Speed effect | Perception |
|-------|------|---|-------------|-----------|
| [METR RCT](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/) | 2025 | 16 experienced devs | **19% slower** | Believed 20% faster |
| [Microsoft "Dear Diary"](https://arxiv.org/abs/2410.18334) | 2025 | 200+ engineers | **No significant change** | 84% reported positive impact |
| [MIT/MS/Accenture](https://mit-genai.pubpub.org/pub/v5iixksv) | 2025 | 1,974 devs | **+26% PR throughput** | — |
| [DORA 2025](https://dora.dev/research/2025/dora-report/) | 2025 | Thousands | +21% individual tasks | **Org delivery flat** |

</div>

<div className="fragment" style={{fontSize: '0.75em', marginTop: '0.5em', backgroundColor: 'rgba(147,112,219,0.15)', padding: '0.8em', borderRadius: '8px'}}>

**The resolution:** Individual speed ≠ organizational throughput. DORA and Faros both found: individuals produce more PRs, but PRs are **154% larger** with **91% longer review times.** The bottleneck shifts from writing to reviewing.

</div>

<aside className="notes">
**The METR perception gap (39 percentage points) is the headline.** But METR later acknowledged serious selection bias and said they "no longer trust their own methodology" for the effect size.

**The MIT study found the opposite:** +26% PR throughput. Crucially, juniors gained 27-39% while seniors gained only 8-13%. AI helps most when you're unfamiliar — exactly the accidental complexity Brooks describes.

**The DORA insight resolves the contradiction:** Individual speed goes up, but org delivery stays flat because the bottleneck shifts to review. More code, same review capacity = same throughput.

## Three Trends Worth Watching Beyond AI

<div style={{fontSize: '0.7em'}}>

<div style={{backgroundColor: 'rgba(200,74,74,0.15)', padding: '0.8em', borderRadius: '8px', marginBottom: '0.5em'}}>

**Memory safety as policy:** The White House called out C and C++ by name. CISA lists memory-unsafe languages as a "bad practice." 45% of organizations now use Rust in production. Microsoft is rewriting Windows kernel components in Rust. Language choice is becoming a regulatory decision. ([L35](/lecture-notes/l35-safety-reliability): safety-by-design embedded in a language.)

<div className="fragment" style={{backgroundColor: 'rgba(50,205,50,0.15)', padding: '0.8em', borderRadius: '8px', marginBottom: '0.5em'}}>

**Formal verification + AI:** Kleppmann argues AI will make formal verification mainstream — not by replacing it, but by making proof generation cheap enough to be practical. If AI generates code we cannot fully review, *proving* correctness becomes more valuable than *testing* for it. Extends [L15](/lecture-notes/l15-testing): unit tests → integration → E2E → formal proofs.

</div>

<div className="fragment" style={{backgroundColor: 'rgba(100,149,237,0.15)', padding: '0.8em', borderRadius: '8px'}}>

[L6](/lecture-notes/l6-immutability-abstraction) (information hiding) at the runtime level. [L21](/lecture-notes/l21-serverless) (sub-millisecond serverless).
